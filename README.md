## Restaurant Menu Management API

Small Ruby on Rails API project for a job application technical test.  
The API manages `Restaurant` and `MenuItem` resources, with basic CRUD, pagination, validation, and tests.

---

## Tech Stack

- **Language**: Ruby 4
- **Framework**: Rails 8 (API-style controllers)
- **Database**: SQLite3 (development & test)
- **Pagination**: Pagy
- **Testing**: Rails default test framework (Minitest) for model & integration tests
- **Deployment**: Railway (using default Rails Dockerfile & GitHub CI config)

---

## Setup Instructions

### 1. Prerequisites

- **Ruby**: 4.x (via `rbenv`/`rvm`/`asdf`)
- **Bundler**: latest
- **SQLite3**
- **Node/Yarn**: only needed if you later add frontend assets; current app is API-focused.

### 2. Clone & Install

```bash
git clone <this-repo-url>
cd restaurant_menu_management

bundle install
```

### 3. Database Setup

```bash
bin/rails db:create
bin/rails db:migrate
bin/rails db:seed
```

This will create the database, run all migrations, and seed:
- At least 2 restaurants
- At least 5 menu items per restaurant

### 4. Run the Server

```bash
bin/rails server
```

The API will be available at `http://localhost:3000`.

Deployed production URL (Railway):
- `https://restaurantmenumanagement-production.up.railway.app/restaurants`

## Testing

Run the test suite with:

```bash

```

There are tests for:
- **Models**: validations and basic behavior
- **Controllers / Integration**: request/response behavior for the main endpoints

## API Overview (High Level)

Resources:
- **Restaurants**
  - CRUD endpoints
  - Each restaurant has many menu items
- **Menu Items**
  - CRUD endpoints
  - Belongs to a restaurant
  - Supports pagination for list endpoints using Pagy

The API follows RESTful conventions as much as possible:
- Standard HTTP verbs (`GET`, `POST`, `PATCH/PUT`, `DELETE`)
- Meaningful HTTP status codes
- JSON responses with clear error messages on validation or runtime errors

## Design Decisions

Below are the main design decisions and the reasoning behind them, ordered for clarity.

1. **Use Ruby on Rails (latest stable)**
   - Chosen to get the **bonus 5% KPI** for using Rails and to get hands-on practice with the same stack used in the target role.
   - Uses **Ruby 4** and **Rails 8**, mainly to start directly on the most recent versions instead of older LTS versions.

2. **Keep controllers clean using concerns**
   - I prefer controllers that stay as thin as possible.
   - I extracted shared/private logic (especially strong parameter handling) into a concern (similar in spirit to Laravel's `FormRequest` pattern that I usually use).
   - This keeps controllers more readable and focused on orchestration instead of low-level parameter handling.

3. **Model-level validations**
   - All field validations live in the **model layer**, following Rails’ recommended best practices from the official docs.
   - This centralizes business rules and constraints, making them easy to test and reuse across different code paths (i believe).

4. **Centralized API exception handling**
   - Implemented an `api_exception_handler` concern (conceptually similar to a global exception handler / trait in PHP).
   - It provides default handling for common exceptions and converts them into consistent JSON responses.
   - Handlers can still be overridden or extended when needed, so this remains flexible while keeping the common cases DRY and practical.

5. **Testing at model and request/integration level**
   - Added **model tests** for validations and basic behavior.
   - Added **integration/controller tests** to cover endpoints and ensure API behavior matches expectations.
   - This aligns with the spec that documentation and tests are a plus, and it ensures changes remain safe.

6. **Pagination with Pagy**
   - For pagination I selected **Pagy**, after a bit of research.
   - Pagy is known for being very fast and lightweight, especially suitable for simple REST APIs.
   - Even if it is not as feature-rich as something like Kaminari, its performance and simplicity are a good fit for this project.

7. **Deployment with default Rails Dockerfile & CI**
   - Used the **default Dockerfile** and **GitHub CI/CD** config generated by Rails.
   - This is sufficient for a small API like this and keeps the deployment setup simple and standard.
   - On Railway, only a few extra configuration values are needed (environment variables, etc.).

8. **Deployment target: Railway**
   - This is my **first time using Railway**, but compared to Fly.io or Render it felt simpler and, importantly, offers a free tier without requiring a credit card.
   - Good fit for a demo/test project where cost and setup friction should be minimal.

9. **Use SQLite for simplicity**
   - The app uses **SQLite3** for both development and test.
   - For a small technical test project, SQLite is more than enough and keeps setup friction low.
   - The codebase can be switched to another RDBMS later by changing the adapter and updating configuration/migrations as needed.

10. **(Planned) Simple token authentication**
    - I considered adding a very simple token-based auth layer for the API.
    - Due to time constraints (Eid al-fitr preparation) this is not implemented yet, but the idea is to add a lightweight token check (e.g. header-based) on protected endpoints.
    - This can be added as a follow-up improvement without changing the current public API behavior.

## Evaluation Criteria Mapping (KPI)

The project is structured with the given KPI in mind:

- **Code quality & structure (25%)**: clean controllers, use of concerns, clear validations, and readable code organization.
- **API design (20%)**: RESTful routes, meaningful HTTP status codes, consistent JSON responses, and centralized error handling.
- **Database modeling (15%)**: proper relations (`Restaurant` has many `MenuItem`), constraints, and migrations reflecting the domain rules.
- **Error handling & edge cases (15%)**: centralized exception handling, validation errors surfaced clearly in responses.
- **Documentation (10%)**: this README aims to be explicit about setup, design choices, and how to work with the project.
- **Git history (10%)**: commits are kept meaningful and incremental instead of using one giant commit.
- **Bonus Rails (5%)**: implemented using Ruby on Rails as requested.
